# Architecture Overview

## Entry Points

- `service/index.js` creates the Express app, mounts middleware, serves static
  files, registers APIs, and starts cron jobs.
- `index.html` is the Vite source HTML entry.
- `src/main.js` owns the browser map app bootstrap.
- `service/app/` is generated by `npm run build` and served by Express.

## Frontend

The frontend is a Vite application:

```text
index.html             Vite HTML entry
src/config.js          frontend runtime config
src/main.js            app bootstrap
src/map/layers.js      Leaflet layer definitions
src/map/location.js    browser/AMap location helpers
src/map/search.js      AMap autocomplete integration
src/map/url-state.js   URL coordinate persistence
src/styles.css         app styles imported by Vite
service/app/           generated static output
```

Only `/` is the supported browser page. The old duplicated `map.html` page and
standalone static scripts were removed.

## Backend

- `service/bin/simpleApi.js` registers explicit API route definitions under
  `/api/v1`.
- `service/bin/service.js` is a thin service layer for API handlers.
- `service/bin/middleware/fetchRelay/index.js` owns tile relay caching.
- `service/bin/whitelist.js` limits relay targets to exact map provider hosts
  and paths.
- `service/bin/cronJob/` contains scheduled maintenance jobs.

## Cache Design

The relay cache stores one binary file plus a JSON metadata sidecar per upstream
URL:

```text
.cache/fetchRelay/<provider>/<url-md5>
.cache/fetchRelay/<provider>/<url-md5>.meta.json
```

The cache is freshness-aware:

- fresh entries are served immediately,
- stale entries are revalidated before serving,
- stale entries may be served only when upstream refresh fails,
- failed or suspicious upstream responses are never cached,
- writes are atomic and metadata-driven.

## Current Stack

- Node.js >= 22.13
- npm package management
- Express 5
- Vite 8
- Leaflet 1.9.4
- AMap JSAPI 2.0 loaded through `@amap/amap-jsapi-loader`
- Node native test runner

## API Direction

The API surface should stay versioned and explicit. New endpoints should be
added under `/api/v1` with:

- a clear route path,
- method restrictions,
- input validation,
- consistent JSON responses,
- route metadata for `/api/v1/routes` and `/api/v1/openapi.json`,
- focused tests when behavior touches cache, validation, or trust boundaries,
- documented behavior in `docs/api.md`.
