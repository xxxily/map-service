# 访问控制加固需求

## 背景

访问控制功能已经支持管理员设置访问密码，并在地图首页显示锁屏，同时拦截未授权瓦片请求。初版实现可以完成主流程，但密码和访问 token 的安全边界需要加固，避免明文持久化、长期密码等价 token 泄露和公开接口暴力尝试。

## 目标

- 访问密码不得以明文形式持久化。
- 访问 token 不得由密码或密码哈希直接充当。
- 访问 token 必须有过期时间，并在密码变更后失效。
- 公开密码验证接口必须具备基础限流能力。
- 首页访问状态检查失败时应默认阻断，而不是默认放行。
- 管理后台应提供清除访问密码的明确入口。

## 范围

### 纳入范围

- 使用带随机 salt 的密码哈希保存访问密码。
- 使用服务端签名访问会话 token，存入 `httpOnly` cookie。
- 移除 URL 查询参数 `access_token` 认证通道。
- `/api/v1/access/verify` 增加失败限流。
- 前端状态检查失败时显示锁屏错误和重试入口。
- 补充访问控制相关单元测试和路由级测试。

### 暂不纳入

- 多用户访问授权。
- 后台可视化登录失败审计。
- 分布式限流存储。
- 与第三方身份系统集成。

## 功能需求

### F1. 访问密码存储

访问密码保存时必须转换为密码哈希。哈希应包含随机 salt，并支持 timing-safe 校验。接口和后台设置返回值不得包含明文密码或密码哈希，只返回 `hasPassword`。这是全新项目，不保留旧版明文配置兼容路径。

### F2. 访问会话 token

访问密码验证成功后，后端创建带过期时间的签名 token，并通过 `map_access_token` 的 `httpOnly` cookie 下发。响应体不得返回 token。

token 必须包含：

- 签发时间
- 过期时间
- 访问控制作用域
- 密码版本或密码更新时间

密码变更或清除后，旧 token 必须失效。

### F3. 瓦片访问拦截

访问控制启用且已设置密码时，`/api/v1/tiles/relay` 必须校验 `map_access_token` cookie。缺少 cookie、cookie 无效或 token 过期时返回 401。

不得再支持通过 URL 查询参数传入访问 token。

### F4. 密码验证限流

`/api/v1/access/verify` 对错误密码请求按客户端维度记录失败次数。超过阈值后在冷却时间内返回 429。验证成功后清除对应失败计数。

### F5. 首页锁屏

地图首页初始化前必须请求 `/api/v1/access/status`。只有服务端明确返回不需要验证或已验证时才初始化地图。状态请求失败时必须显示锁屏错误和重试入口，不得默认初始化地图。

### F6. 管理后台交互

访问控制设置应支持：

- 启用访问密码。
- 设置或修改访问密码。
- 清除已保存访问密码。

启用访问控制时，如果没有已保存密码且没有输入新密码，应拒绝保存。

## 非功能需求

- 新访问密码长度至少 10 位。
- 密码和 token 比较使用 timing-safe 方式。
- 不在日志中输出明文密码、密码哈希或访问 token。
- `httpOnly` cookie 使用 `sameSite: 'lax'`。HTTPS 请求下应设置 `secure: true`。
- 限流默认可使用进程内存储；未来如多实例部署，再升级为共享存储。

## API 说明

| 方法 | 路径 | 行为 |
| --- | --- | --- |
| `GET` | `/api/v1/access/status` | 返回 `{ required: boolean }` |
| `POST` | `/api/v1/access/verify` | 验证密码，成功时设置 `map_access_token` cookie |
| `GET` | `/api/v1/tiles/relay` | 访问控制启用时只接受有效 cookie 授权 |

## 验收标准

- `settings.json` 中不再保存 `access.password` 明文。
- `/access/verify` 成功响应体不包含 token。
- URL 参数 `access_token` 不能绕过瓦片拦截。
- 错误密码连续超过阈值后返回 429。
- 状态检查失败时首页不初始化地图。
- 密码变更后旧访问 token 失效。
- `npm run check`、`npm test`、`npm run build` 通过。
