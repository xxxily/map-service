# 功能完备地图应用建设需求

## 背景和目标

当前 `map-service` 已经具备地图应用的基础底座：

- 前台地图基于 Leaflet，支持高德和 Google 瓦片底图、旋转视角、定位、地点搜索、URL 视图状态和图层记忆。
- KML 能在浏览器本地导入、渲染点线面、编辑点位属性、拖拽点位、导出标准 KML，并处理 WGS84 与高德 GCJ-02 纠偏。
- 后端提供 `/api/v1` 版本化接口，支持瓦片白名单代理、服务端缓存、访问密码、管理后台登录、代理策略、图层目录、预缓存任务和访问统计。
- 管理后台已经覆盖系统概览、缓存管理、预缓存、图层查看、代理和访问控制设置。
- PWA 已缓存应用壳和静态资源，离线时能提供基础兜底页。

这些能力更接近“地图展示 + 瓦片代理缓存 + 运维后台”。如果要把项目打造成功能完备的地图应用，需要继续补齐用户侧地图工作流、数据持久化、离线能力、业务协作、可配置图层体系和可观测运维能力。

本需求文档作为总纲，用于梳理后续产品能力和系统改造方向。具体模块进入实现前，可以再拆分为更细的专项需求文档。

## 产品定位

目标产品应从单人本地地图工具升级为可部署、可管理、可持续演进的地图应用平台：

- 普通用户能快速浏览地图、搜索地点、定位自己、收藏地点、测量距离面积、分享视图。
- 外业或业务用户能管理点线面数据、导入导出常见地理数据格式、离线使用地图和标注。
- 管理员能配置底图、管理缓存、预热离线区域、设置访问控制、观察服务健康和用户访问情况。
- 开发者能基于稳定 API、模块化前端和清晰数据模型继续扩展业务能力。

## 用户和场景

### U1. 普通地图浏览用户

用户目标：

- 打开地图后快速看到上次位置或默认城市。
- 搜索地点并定位到 POI。
- 获取当前位置，查看附近地物。
- 切换街道、卫星、混合底图。
- 把当前地图位置分享给其他人。

当前支持：

- 基础地图浏览、底图切换、搜索、定位、URL `coords` 分享和本地视图记忆。

主要缺口：

- 搜索结果缺少列表、详情、历史、收藏和错误恢复。
- URL 状态只记录中心点、缩放和方向，无法表达图层、选中要素、KML 数据集或测量状态。
- 移动端工具菜单和地图控件仍偏基础，缺少面向高频使用的交互整合。

### U2. 外业采集和标注用户

用户目标：

- 在地图上新增点、线、面。
- 编辑名称、描述、分类、颜色、图标和附加字段。
- 批量导入外部数据，修正位置后导出。
- 离线环境中继续查看底图和业务数据。

当前支持：

- KML 导入、点线面渲染、点位新增、点位拖拽、要素名称和描述编辑、导出 KML。

主要缺口：

- 数据只保存在浏览器 localStorage，换设备、换浏览器或清缓存后丢失。
- 只能新增点，不能完整绘制线和面。
- 不支持要素样式、分类、附件、照片、时间字段和自定义属性。
- 不支持 GeoJSON、GPX、CSV、KMZ 等常见格式。
- 不支持数据集版本、撤销/重做、批量编辑和冲突处理。

### U3. 离线和弱网用户

用户目标：

- 在网络不稳定时仍能打开地图应用。
- 提前下载指定区域底图和业务数据。
- 明确知道离线包范围、大小、有效期和下载进度。
- 离线新增或编辑的数据能在恢复网络后同步。

当前支持：

- PWA 应用壳离线兜底。
- 后台可创建服务端瓦片预缓存任务。

主要缺口：

- 前台用户不能直接管理个人离线区域。
- Service Worker 不缓存 `/api/` 请求和瓦片，客户端离线地图不可用。
- 服务端预缓存和客户端离线包没有统一的数据模型。
- 不支持离线编辑队列和恢复网络后的同步策略。

### U4. 管理员和运维人员

用户目标：

- 配置底图、代理、访问控制和默认地图行为。
- 观察缓存命中、预缓存进度、错误来源和服务健康。
- 控制存储占用，避免误下载超大瓦片任务。
- 审计敏感操作。

当前支持：

- 管理后台登录、系统概览、缓存状态、访问统计、代理设置、图层目录、预缓存任务和访问密码。

主要缺口：

- 管理后台缺少审计日志、任务重试策略、失败详情聚合、实时进度推送和告警。
- 图层目录仍在代码中维护，前台图层配置和后台 provider 目录存在重复表达。
- 缓存策略缺少按图层、区域、时间、大小上限的可视化治理。
- 管理员权限仍是单账号模式，没有角色、操作边界和会话管理。

### U5. 数据管理员和开发者

用户目标：

- 维护数据集、图层目录、样式规则和 API 集成。
- 通过稳定接口接入第三方系统。
- 在测试、开发、生产环境中使用不同密钥和默认配置。

当前支持：

- `/api/v1` 路由目录和轻量 OpenAPI。
- 后台设置持久化在 `.db/admin/`。

主要缺口：

- 高德 JSAPI key 和 securityJsCode 写在前端配置中，缺少环境化配置和密钥轮换方案。
- 地图数据缺少服务端数据模型和 API。
- 缺少插件化图层、业务图层和样式配置体系。
- 缺少端到端测试、可访问性测试和移动端视觉回归验证。

## 范围

### 范围内

- 完善前台地图浏览、搜索、定位、图层、分享、测量、标注和数据管理体验。
- 建立服务端地图数据模型，让 KML/标注/收藏/离线区域不再只依赖 localStorage。
- 打通前台离线区域和后台预缓存能力。
- 将底图和业务图层配置统一为可配置目录。
- 增强管理后台的可观测、审计、权限和缓存治理能力。
- 梳理前后端模块边界，为后续专项实现提供路线图。

### 范围外

- 不自研完整商业导航引擎。
- 不自建全球地理编码、POI 和路径规划数据源。
- 不替代专业 GIS 桌面软件的复杂空间分析能力。
- 不在第一阶段支持多实例分布式瓦片调度和大规模 GIS 数据仓库。

## 功能需求

### F1. 地图首页与基础浏览

- 地图首页应清晰区分地图画布、工具栏、搜索区、图层区和数据区。
- 首次进入时使用管理员配置的默认中心点、缩放级别、默认图层和语言。
- 再次进入时优先恢复用户最近视图，但提供一键回到默认视图。
- URL 分享应支持中心点、缩放、方向、当前底图、可选选中要素和可选业务图层。
- 地图控件应适配桌面和移动端，不应遮挡 Leaflet attribution、缩放控件、搜索结果和 KML 面板。
- 地图加载失败、搜索 SDK 加载失败、瓦片 401/403/5xx 时应有明确的非阻塞提示。

### F2. 搜索与地点能力

- 搜索输入应提供结果列表，而不是只依赖高德控件的选择事件。
- 结果列表至少展示名称、地址、类型、距离当前视图中心的距离。
- 支持搜索历史、本地清除历史和快速再次搜索。
- 支持收藏地点，收藏应持久化到服务端账号或访问会话下。
- 支持点击结果后在地图上显示临时标记和详情卡片。
- 支持从详情卡片创建正式标注点。
- 搜索 SDK 不可用时，应降级为经纬度输入和本地数据集搜索。

### F3. 定位与位置跟随

- 定位按钮应显示定位中、成功、失败、权限被拒绝和低精度状态。
- 定位成功后展示当前位置、精度半径和更新时间。
- 支持一次性定位和连续跟随两种模式。
- 支持复制当前坐标，并明确显示坐标系。
- 坐标转换失败时保留原始 GPS 坐标，并提示可能存在偏移。
- 移动端应避免定位提示和工具栏重叠。

### F4. 图层体系

- 前台底图、后台 provider 和预缓存 provider 应统一来自同一个图层目录。
- 图层目录应支持运行时配置，至少包括：
  - 图层 ID、名称、类型、厂商、URL 模板、子域、最大最小缩放。
  - 是否可作为底图、是否可叠加、默认透明度、默认代理策略。
  - 版权/attribution、服务条款说明和可用区域。
- 支持新增、编辑、禁用自定义 XYZ/TMS 瓦片图层。
- 支持叠加图层透明度调整和顺序调整。
- 支持图层健康检查，显示最近请求状态、缓存命中和失败原因。
- 前台图层切换应保存用户偏好，并能通过 URL 分享。

### F5. 标注、绘制与业务数据

- 支持创建和编辑点、线、面三类要素。
- 点要素支持拖拽；线和面支持顶点编辑、插入顶点和删除顶点。
- 要素属性至少包括名称、描述、分类、颜色、图标、创建时间、更新时间。
- 支持自定义字段，字段类型至少包括文本、数字、日期、单选、多选和链接。
- 支持附件或照片的元数据预留；文件存储可作为后续专项实现。
- 支持撤销/重做当前编辑会话。
- 支持批量选择、批量删除、批量修改分类或样式。
- 支持要素过滤、关键字搜索和按分类显示/隐藏。
- 支持点击地图空白处退出编辑态，避免误操作。
- 所有删除和覆盖操作必须使用项目统一弹窗组件确认，不得使用原生阻塞弹窗。

### F6. 数据集与导入导出

- 引入“数据集”概念，KML 文件、手工标注和未来业务图层都归属于数据集。
- 数据集应支持命名、描述、显示开关、默认样式、可编辑状态和更新时间。
- KML 导入导出继续保持 WGS84 标准坐标，不因底图纠偏污染原始数据。
- 在 KML 基础上新增 GeoJSON 导入导出，作为前后端通用空间数据格式。
- 后续支持 CSV 点位导入、GPX 轨迹导入和 KMZ 解压导入。
- 导入时应展示预览摘要，包括要素数量、几何类型、坐标范围和无法识别的记录。
- 导入大文件时应避免阻塞主线程，必要时引入 Web Worker。
- 导出时支持选择数据集、要素范围、坐标系和文件格式。

### F7. 测量与分析工具

- 支持距离测量，能显示分段距离和总距离。
- 支持面积测量，能显示周长和面积。
- 支持清除当前测量结果和将测量结果保存为数据集要素。
- 支持坐标拾取，点击地图后展示经纬度、缩放级别和可复制文本。
- 支持比例尺显示。
- 后续可扩展缓冲区、点到线距离、区域内要素统计等轻量分析能力。

### F8. 离线地图与弱网体验

- 前台应提供“离线区域”入口，用户可以基于当前视野创建离线包。
- 离线包创建前展示区域、图层、缩放级别、预计瓦片数和预计大小。
- 离线包应复用后台预缓存的估算逻辑和瓦片范围算法。
- 客户端离线包应有本地状态：等待中、下载中、已完成、失败、过期。
- Service Worker 应明确区分应用壳、瓦片缓存和业务数据缓存。
- 离线编辑应进入本地变更队列，恢复网络后提示同步。
- 管理后台可以查看服务端预缓存和用户离线包占用的存储情况。

### F9. 分享与协作

- 支持复制当前视图链接。
- 支持复制单个要素链接。
- 支持导出静态截图或打印视图，至少包含地图、比例尺、时间和图层说明。
- 支持只读分享和可编辑分享两种权限模型的预留。
- 多用户协作不作为第一阶段目标，但数据模型应避免只能绑定单浏览器。

### F10. 管理后台增强

- 增加审计日志，记录登录、退出、修改设置、清理缓存、创建/暂停/删除预缓存任务等操作。
- 增加任务失败详情聚合，按 provider、HTTP 状态、网络错误和白名单错误分类。
- 增加任务重试策略，支持失败瓦片重试和仅重试失败项。
- 增加任务实时进度推送，可选 SSE 或 WebSocket。
- 增加缓存治理能力：
  - 按 provider、图层、创建时间、最后访问时间和大小清理。
  - 设置全局缓存大小上限和自动淘汰策略。
  - 查看单个 URL 的缓存元数据和回源状态。
- 增加管理会话列表和主动失效能力。
- 后续支持多管理员和角色权限：系统管理员、运维人员、数据管理员、只读观察者。

### F11. 配置与部署

- 高德 key、securityJsCode、默认地图视图、默认图层应支持环境变量或后端配置下发。
- 前端不应将生产密钥硬编码在源码中；生产部署应支持密钥轮换。
- 支持按环境区分开发、测试、生产配置。
- 文档应补充地图服务商密钥申请、域名白名单和服务条款注意事项。
- 服务启动时应校验关键配置，并在管理后台展示配置健康状态。

## 系统改造需求

### A1. 前端模块化改造

当前地图入口主要集中在 `src/main.js`，KML、搜索、定位和图层模块已有拆分，但仍由入口统一串联。后续应演进为更清晰的地图应用架构：

- `src/map/app.js`：地图应用编排层，负责初始化和模块生命周期。
- `src/map/state.js`：地图运行状态，包括视图、图层、选择要素、编辑模式。
- `src/map/tools/`：搜索、定位、测量、绘制、分享等工具模块。
- `src/map/data/`：数据集、导入导出、要素管理和本地缓存。
- `src/map/layers/`：底图和叠加图层管理。
- `src/map/offline/`：客户端离线包管理。

改造时应保持现有功能可用，优先抽离状态和数据模型，再逐步迁移 UI。

### A2. 服务端数据模型

需要从浏览器 localStorage 迁移到服务端持久化。第一阶段可继续使用 `.db/` JSON 文件或轻量嵌入式存储，后续再评估 SQLite/Postgres。

建议核心模型：

- `MapUser`：用户或访问主体。
- `MapProject`：地图项目或工作空间。
- `MapDataset`：数据集，包含名称、描述、类型、样式、权限和可见性。
- `MapFeature`：空间要素，使用 GeoJSON Geometry 保存几何，properties 保存业务属性。
- `MapLayer`：底图或叠加图层配置。
- `OfflineArea`：离线区域或服务端预缓存区域。
- `AuditLog`：管理和数据变更审计日志。

### A3. API 分层

新增用户侧地图 API 应继续挂载在 `/api/v1` 下，并使用统一 JSON 响应。

建议新增 API 分组：

| 分组 | 示例路径 | 用途 |
| --- | --- | --- |
| 地图配置 | `GET /api/v1/map/config` | 获取默认视图、前端可用图层和功能开关 |
| 图层 | `GET /api/v1/map/layers` | 获取前台可用图层目录 |
| 数据集 | `GET /api/v1/map/datasets` | 列出数据集 |
| 数据集 | `POST /api/v1/map/datasets` | 创建数据集 |
| 要素 | `GET /api/v1/map/datasets/:id/features` | 获取数据集要素 |
| 要素 | `POST /api/v1/map/datasets/:id/features` | 创建要素 |
| 要素 | `PUT /api/v1/map/features/:id` | 更新要素属性或几何 |
| 要素 | `DELETE /api/v1/map/features/:id` | 删除要素 |
| 导入导出 | `POST /api/v1/map/import` | 上传并解析 KML/GeoJSON |
| 导入导出 | `GET /api/v1/map/export` | 导出选定数据 |
| 离线 | `POST /api/v1/map/offline/estimate` | 估算离线区域 |
| 离线 | `POST /api/v1/map/offline/areas` | 创建离线区域 |
| 审计 | `GET /api/v1/admin/audit-logs` | 查看管理审计日志 |

### A4. 权限边界

- 当前访问密码适合“私有地图入口保护”，但不足以表达数据编辑权限。
- 后续应区分：
  - 访问地图应用。
  - 查看数据集。
  - 编辑数据集。
  - 管理系统设置。
  - 执行缓存和预缓存操作。
- 在多用户实现前，至少应在 API 设计上预留 `ownerId`、`createdBy`、`updatedBy`、`visibility` 和 `permission` 字段。

### A5. 本地存储迁移

现有本地数据包括：

- `last_map_view`
- `last_map_layer`
- `map_kml_list`
- `mapServiceAdminToken`

迁移策略：

- 视图和图层偏好可以继续使用 localStorage，但应增加版本号和容错。
- `map_kml_list` 应提供一次性迁移入口，允许用户导入到服务端数据集。
- 管理 Token 保持 localStorage 前应评估 XSS 风险；后续可改为 httpOnly cookie 管理会话。
- 所有本地存储读取都需要 try/catch，并避免数据损坏导致应用无法启动。

### A6. 可测试性和质量门禁

- 新增后端 API 必须补充 Node 原生测试。
- 关键前端纯函数应抽出并补充单元测试，例如 URL 状态解析、图层 URL 构造、GeoJSON/KML 转换、瓦片估算。
- 地图 UI 变更应补充至少一条浏览器级验证脚本或手动验证清单。
- 继续执行提交前门禁：`npm run check`、`npm test`、`npm run build`。

## 非功能需求

### 性能

- 首屏应尽快展示地图壳，不应被 KML 大文件解析或后台数据加载阻塞。
- 大数据集渲染需要聚合、抽稀或按视窗加载，避免一次性渲染过多 DOM/SVG 图层。
- 瓦片请求应尊重并发和缓存策略，避免对上游造成突发压力。
- 预缓存任务必须保留瓦片数量上限和并发上限。

### 可用性

- 地图核心操作应可在桌面和移动端完成。
- 操作反馈应使用非阻塞提示、统一确认弹窗和明确的 loading 状态。
- 工具按钮需要有 `aria-label` 或 tooltip。
- 表单错误应定位到具体字段。

### 安全

- 不在源码中提交生产地图密钥、代理密码、访问密码或 token secret。
- 所有写操作 API 必须鉴权并校验输入。
- 导入文件必须限制大小和格式，解析时防止 XML 外部实体和恶意内容。
- 导出的 HTML 或 popup 内容必须转义，避免 XSS。
- 审计日志不得记录明文密码、token 和代理密码。

### 可靠性

- 服务重启后预缓存任务状态应可恢复或明确标记中断。
- 离线包和缓存清理应避免删除正在使用的任务文件。
- 配置写入应保持原子性，避免半写入导致启动失败。
- 图层上游失败时应尽量使用 stale 缓存兜底，并在 UI 标识缓存状态。

### 合规

- 图层目录必须记录 attribution 和服务条款说明。
- 离线下载和预缓存能力需要遵守地图服务商条款，管理后台应提示风险。
- 用户上传的地理数据应明确存储位置和清理方式。

## 验收标准

### 产品验收

- 用户无需查看说明即可完成：打开地图、搜索地点、定位当前位置、切换底图、复制分享链接。
- 用户可以创建点线面数据，编辑属性和样式，并在刷新页面后继续使用。
- 用户可以导入 KML 和 GeoJSON，查看导入摘要，修正数据后导出。
- 用户可以创建离线区域，并在弱网或离线场景下打开应用和查看已缓存区域。
- 管理员可以新增或禁用图层，并让前台和预缓存同时生效。
- 管理员可以查看预缓存任务失败原因、重试失败项并审计关键操作。

### 工程验收

- 新增需求对应的 API 均位于 `/api/v1`，并更新 `docs/api.md`。
- 前端业务代码不直接使用 `alert`、`confirm`、`prompt`。
- 不手动编辑 `service/app/` 构建产物，前端源码修改后通过 `npm run build` 生成。
- `npm run check`、`npm test`、`npm run build` 通过。
- 新增数据模型、迁移脚本或运行时状态文件在文档中说明存储位置和备份策略。

## 风险和待决策事项

- 地图服务商条款可能限制瓦片预缓存和离线下载，需要先确认允许范围。
- 如果要支持多用户和协作，需要选择长期存储方案；JSON 文件只适合早期单实例。
- 前端继续使用 Leaflet 可以满足当前二维地图需求，但大规模矢量数据和复杂样式可能需要评估 MapLibre GL。
- 生产环境密钥下发方式需要在安全性、部署便利性和前端 SDK 限制之间取舍。
- 客户端离线瓦片缓存会显著增加浏览器存储占用，需要配额检测和清理策略。
