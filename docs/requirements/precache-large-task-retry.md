# 预缓存大任务与失败重试优化需求

## 背景

管理后台预缓存任务当前把 `maxTiles` 作为硬性创建上限。实际使用中，大区域任务可能超过建议上限，但管理员仍需要创建，只应收到风险提示。同时，连续高速请求地图服务商可能触发限流或临时错误，任务需要具备节流、失败记录、自动退避和手动重试能力。

## 目标

- 超过建议瓦片数量的任务允许创建，后台只展示风险提示。
- 预缓存任务支持请求间隔，默认 0 毫秒，在并发基础上控制请求发起速度。
- 失败瓦片要记录为可重试数据，不能因为一次错误永久失去更新能力。
- 失败记录需要有上限；任务错误过多时应主动暂停，等待用户手动继续。
- 识别 429 等限流错误后，任务应自动等待并重试；多次限流后暂停，避免持续消耗系统和上游资源。

## 行为约定

- `maxTiles` 是建议上限，不再阻止 `POST /api/v1/admin/precache/tasks` 创建任务。
- `POST /api/v1/admin/precache/estimate` 继续返回 `withinLimit: false`，由前端展示警告。
- 新增任务参数 `requestIntervalMs`，表示任意两个瓦片请求开始之间的最小间隔，默认 0。
- 任务保存 `failedTiles`，每个失败瓦片按 `z/x/y` 去重记录最近错误、状态码、失败时间和尝试次数。
- `failed` 表示当前仍未成功的失败瓦片数量；重试成功后会减少。
- `completed_with_errors`、`failed`、`paused`、`interrupted` 任务都可以通过继续操作重试失败瓦片。
- 当失败数量达到任务阈值，或 429 自动退避重试次数耗尽，任务自动暂停并保留失败明细。

## 非目标

- 不引入独立任务队列服务。
- 不实现跨进程并发调度。
- 不改变现有瓦片缓存目录结构。
