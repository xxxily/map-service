# KML 公共图层共享管理需求

## 背景

当前 KML 功能完全基于浏览器端 localStorage 运作：用户在前台导入或新建 KML 文件，数据保存在本地浏览器中，换设备或清缓存即丢失。这种模式适合个人轻量标注，但无法满足以下场景：

- 管理员整理了一批公共地理标注（如设备点位、巡检路线、区域边界），希望共享给所有访问地图的用户。
- 多人需要查看同一组 KML 数据，但每个人都要自己导入一遍，且无法保持一致。
- 管理员更新了某个 KML 文件后，用户端无法感知变化。
- 当前没有任何服务端 KML 存储，管理后台也没有 KML 管理模块。

## 目标

1. **管理员**可以在前台地图上以正常工作流导入或新建 KML，然后选择将其共享为公共图层。
2. **管理员**可以在后台管理模块查看、启用/禁用、重命名和删除公共 KML。
3. **管理员**可以随时取消共享或临时禁用某个公共 KML，禁用后前台不再向用户展示。
4. **管理员**可以从后台跳转到前台地图编辑公共 KML 内容，调整完成后重新发布。
5. **普通用户**在前台地图可以发现当前有哪些可用的公共 KML，并按需启用加载到地图上。
6. **普通用户**的个人 KML 功能不受影响，继续使用现有的 localStorage 工作流。

## 用户角色

### 管理员

已登录后台的运维或数据管理员。拥有公共 KML 的完整管理权限：创建、编辑、发布、禁用、取消共享、删除。

### 普通用户

通过访问密码或无密码方式访问前台地图的用户。只能浏览和按需启用公共 KML，不能编辑公共 KML 内容。个人 KML 功能与现有行为一致。

## 范围

### 纳入范围

- 后端公共 KML 数据持久化。
- 后台新增"KML 管理"页面。
- 前台新增公共 KML 发现和加载入口。
- 管理员在前台发布个人 KML 为公共图层。
- 管理员在前台编辑公共 KML 并重新发布。
- 公共 KML 的生命周期管理：草稿、已发布、已禁用。
- KML 文件上传（.kml 文件直接上传到服务端）。

### 暂不纳入

- 公共 KML 的多管理员协同编辑和版本冲突处理。
- 公共 KML 的细粒度权限（按用户组可见）。
- GeoJSON、GPX、KMZ 等其他格式的公共图层。
- 公共 KML 的实时推送更新（用户需手动刷新或重新加载）。
- 公共 KML 的访问统计和审计日志。

## 核心概念

### 个人 KML

与当前行为一致，数据保存在用户浏览器 localStorage，只对当前浏览器可见。管理员和普通用户都可以操作个人 KML。

### 公共 KML

由管理员发布到服务端的 KML 数据，所有前台用户可以发现并加载。公共 KML 保存在服务端 `.db/admin/` 目录下，随服务端持久化。

### 公共 KML 状态

| 状态 | 说明 | 前台用户是否可见 |
| --- | --- | --- |
| `draft` | 草稿，管理员正在编辑但尚未发布 | 不可见 |
| `published` | 已发布，前台用户可以发现和加载 | 可见 |
| `disabled` | 管理员临时禁用，不影响数据 | 不可见 |

## 功能需求

### F1. 后台 KML 管理页面

后台导航新增"KML 管理"标签页，路由为 `/admin/kml`。

KML 管理页面展示所有公共 KML 列表，每个条目显示：

- 名称
- 状态标签（草稿 / 已发布 / 已禁用）
- 要素数量
- 坐标纠偏模式
- 创建时间
- 最后更新时间

#### 后台操作

| 操作 | 说明 |
| --- | --- |
| 导入 KML 文件 | 上传 .kml 文件，解析后创建为公共 KML（默认草稿状态） |
| 新建空白 KML | 创建一个空的公共 KML（默认草稿状态） |
| 重命名 | 修改公共 KML 名称 |
| 发布 | 将草稿或已禁用的 KML 设为已发布，前台用户可见 |
| 禁用 | 将已发布的 KML 设为已禁用，前台用户不可见 |
| 删除 | 永久删除公共 KML 数据，需二次确认 |
| 编辑内容 | 跳转到前台地图，加载该公共 KML 进行编辑 |
| 切换纠偏 | 开启或关闭坐标纠偏 |
| 导出 | 将公共 KML 导出为标准 .kml 文件下载 |

### F2. 管理员前台发布个人 KML 为公共图层

管理员已登录后台时，前台 KML 面板中每个个人 KML 文件卡片增加"共享为公共 KML"操作按钮（仅管理员登录态下可见）。

点击后：

1. 将个人 KML 的要素数据上传到服务端，创建为公共 KML。
2. 默认状态为 `published`（直接发布），管理员也可以选择保存为草稿。
3. 发布成功后提示成功，个人 KML 本身不受影响（两者独立）。

注意：发布后公共 KML 与原个人 KML 相互独立，没有双向同步关系。管理员后续编辑公共 KML 不会影响个人 KML，反之亦然。

### F3. 管理员前台编辑公共 KML

管理员可以从后台 KML 管理页面点击"编辑内容"跳转到前台地图。

跳转时通过 URL 参数传递公共 KML ID，例如 `/?editPublicKml=<id>`。前台地图识别到该参数后：

1. 从服务端加载指定公共 KML 的完整数据。
2. 渲染到地图上，进入公共 KML 编辑模式。
3. 管理员可以正常新增、删除、拖拽点位，编辑要素属性。
4. 编辑模式下提供"保存"和"保存并发布"按钮：
   - "保存"：将当前编辑结果保存回服务端，保持原有状态。
   - "保存并发布"：保存并将状态设为 `published`。
5. 编辑模式下提供"退出编辑"按钮，未保存的修改需要确认是否放弃。

编辑公共 KML 时不应影响个人 KML 列表和 localStorage 数据。

### F4. 前台公共 KML 发现与加载

前台 KML 面板新增"公共 KML"分区（位于个人 KML 列表上方或下方），展示所有状态为 `published` 的公共 KML。

列表中每个公共 KML 条目显示：

- 名称
- 要素数量
- 一个启用/禁用开关

#### 交互规则

- 公共 KML 列表默认折叠，点击"公共 KML"标题栏展开。
- 公共 KML 默认禁用（不显示在地图上），用户可以按需启用。
- 用户启用公共 KML 后，从服务端拉取完整数据并渲染到地图上。
- 用户的启用/禁用偏好保存在 localStorage，避免每次刷新都重置。
- 普通用户不能编辑公共 KML 的内容（点位不可拖拽，弹窗不显示编辑/删除按钮）。
- 普通用户可以导出公共 KML 为标准 .kml 文件（只读导出）。
- 公共 KML 要素的 Popup 和 Tooltip 与个人 KML 保持一致的样式，但不显示编辑和删除操作。

#### 缓存与刷新

- 用户启用公共 KML 时从服务端加载最新数据。
- 已加载的公共 KML 可以保留在内存中，不需要每次交互都重新请求。
- 公共 KML 列表的元信息（名称、要素数量）在打开 KML 面板时从服务端刷新。
- 前台公共 KML 面板提供一个手动刷新按钮，点击后重新拉取列表和已加载的 KML 数据。

### F5. 管理员前台身份识别

前台需要识别当前用户是否为已登录管理员，以决定是否显示"共享为公共 KML"和编辑公共 KML 等管理功能。

识别方式：检测 localStorage 中是否存在有效的管理员 Token（复用现有 `mapServiceAdminToken`），并在需要时通过 `GET /api/v1/admin/session` 校验 Token 有效性。

- Token 有效：展示管理员专属操作按钮。
- Token 无效或不存在：只展示普通用户功能。
- Token 校验失败时不应阻断正常地图使用，静默降级为普通用户模式。

## 数据模型

### 公共 KML 存储

公共 KML 数据存储在 `.db/admin/shared-kml.json`，复用现有 `AdminStore` 机制。

```json
[
  {
    "id": "shared-kml-1719561600000-a1b2c3",
    "name": "设备巡检点位",
    "status": "published",
    "coordCorrection": "wgs84-to-gcj02",
    "features": [
      {
        "id": "feat-1719561600001-d4e5f6",
        "type": "Point",
        "name": "1 号基站",
        "description": "位于某某路口",
        "coordinates": [113.264, 23.129]
      }
    ],
    "createdAt": "2026-06-28T08:00:00.000Z",
    "updatedAt": "2026-06-28T08:00:00.000Z"
  }
]
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 唯一 ID，格式 `shared-kml-<timestamp>-<random>` |
| `name` | string | 是 | 公共 KML 名称 |
| `status` | string | 是 | 状态：`draft` / `published` / `disabled` |
| `coordCorrection` | string | 是 | 坐标纠偏模式：`wgs84-to-gcj02` / `none` |
| `features` | array | 是 | 要素数组，结构与现有个人 KML 一致 |
| `createdAt` | string | 是 | ISO 8601 创建时间 |
| `updatedAt` | string | 是 | ISO 8601 最后更新时间 |

### 前台用户本地偏好

用户对公共 KML 的启用/禁用偏好保存在 localStorage：

```json
{
  "shared-kml-1719561600000-a1b2c3": true,
  "shared-kml-1719561600000-x7y8z9": false
}
```

存储 key：`map_shared_kml_prefs`。

## API 设计

### 公开接口（无需管理员鉴权）

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/kml/shared` | 获取所有已发布的公共 KML 摘要列表 |
| `GET` | `/api/v1/kml/shared/:id` | 获取指定公共 KML 的完整数据 |

#### `GET /api/v1/kml/shared`

返回所有 `status === 'published'` 的公共 KML 摘要：

```json
{
  "code": 0,
  "result": [
    {
      "id": "shared-kml-1719561600000-a1b2c3",
      "name": "设备巡检点位",
      "featureCount": 12,
      "coordCorrection": "wgs84-to-gcj02",
      "updatedAt": "2026-06-28T08:00:00.000Z"
    }
  ]
}
```

摘要不包含 `features` 数组，减小响应体积。包含 `featureCount` 用于前台展示。

#### `GET /api/v1/kml/shared/:id`

返回指定公共 KML 的完整数据（包含 `features`）。只返回 `status === 'published'` 的 KML，其他状态返回 404。

如果前台访问控制已启用，这两个接口需要和瓦片接口一样校验访问 cookie。

### 管理接口（需管理员鉴权）

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/v1/admin/kml` | 获取所有公共 KML 列表（含所有状态） |
| `GET` | `/api/v1/admin/kml/:id` | 获取指定公共 KML 完整数据（含所有状态） |
| `POST` | `/api/v1/admin/kml` | 创建公共 KML |
| `PUT` | `/api/v1/admin/kml/:id` | 更新公共 KML（名称、状态、纠偏、要素） |
| `DELETE` | `/api/v1/admin/kml/:id` | 删除公共 KML |

#### `POST /api/v1/admin/kml`

创建公共 KML。请求体：

```json
{
  "name": "设备巡检点位",
  "status": "published",
  "coordCorrection": "wgs84-to-gcj02",
  "features": []
}
```

`status` 缺省为 `draft`。`coordCorrection` 缺省为 `wgs84-to-gcj02`。

#### `PUT /api/v1/admin/kml/:id`

部分更新公共 KML。只传需要变更的字段，未传字段保持不变：

```json
{
  "name": "新名称",
  "status": "disabled"
}
```

更新 `features` 时传完整数组覆盖（不支持单要素增删接口，保持简单）。每次更新自动刷新 `updatedAt`。

#### `DELETE /api/v1/admin/kml/:id`

永久删除公共 KML 数据。返回 204 或统一 JSON。

### 文件上传接口

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `POST` | `/api/v1/admin/kml/import` | 上传 KML 文件并创建公共 KML |

接收 `multipart/form-data`，字段名 `file`。解析 KML 文件提取要素，创建公共 KML。可选 form 字段：

- `name`：覆盖文件名作为 KML 名称。
- `status`：创建后的状态，默认 `draft`。
- `coordCorrection`：坐标纠偏模式，默认 `wgs84-to-gcj02`。

文件大小限制：10MB。

## 交互设计

### 后台 KML 管理页面

- 页面顶部提供"导入 KML 文件"和"新建空白 KML"两个主操作按钮。
- KML 列表使用卡片或表格形式展示。
- 每个条目右侧提供操作按钮组：发布/禁用切换、重命名、编辑内容、导出、删除。
- 状态使用不同颜色标签区分：草稿（灰色）、已发布（绿色）、已禁用（橙色）。
- 编辑内容按钮使用新窗口或当前窗口跳转到前台地图，携带 `editPublicKml` 参数。
- 删除操作使用项目统一弹窗组件确认。

### 前台公共 KML 分区

- KML 面板中在个人 KML 列表上方增加"公共图层"分区。
- 分区标题栏显示"公共图层"文字和当前已发布数量，点击可折叠/展开。
- 分区标题栏右侧提供刷新按钮。
- 每个公共 KML 条目显示名称、要素数量和启用/禁用开关。
- 启用/禁用开关使用 toggle 样式，与现有 KML 文件的显隐控制视觉一致。
- 无已发布的公共 KML 时，分区内显示"暂无公共图层"提示文字。

### 管理员前台共享按钮

- 仅在检测到有效管理员 Token 时显示。
- 位于个人 KML 文件卡片的操作区域，使用共享/上传图标。
- 点击后弹出确认弹窗，可选择发布状态（直接发布 / 保存为草稿）。
- 共享成功后显示成功提示。

## 非功能需求

### 性能

- 公共 KML 摘要列表接口应轻量，不返回 features 数组。
- 大量要素的公共 KML（超过 500 个要素）在前台加载时应显示 loading 状态。
- 公共 KML 渲染与个人 KML 共享同一个 Leaflet 地图实例，注意图层数量对渲染性能的影响。

### 安全

- 管理接口严格校验管理员 Token。
- 公开的 KML 共享接口在访问控制启用时需校验访问 cookie。
- KML 文件上传限制大小和格式，防止恶意文件。
- 要素名称和描述在前端渲染时必须转义 HTML，防止 XSS。
- 服务端解析 KML XML 时注意 XXE 防护（复用现有 parseKML 的客户端解析逻辑在服务端运行时需要使用安全的 XML 解析器）。

### 可靠性

- 公共 KML 数据写入使用 AdminStore 的原子写入机制。
- 前台加载公共 KML 失败时显示错误提示，不影响个人 KML 和地图正常使用。
- 管理员编辑公共 KML 时，未保存即离开需要确认提示。

### 一致性

- 后台和前台操作同一份服务端数据，不存在双写。
- 管理员在后台禁用某个公共 KML 后，前台新打开页面不再展示该 KML；已加载到地图上的不主动移除，用户下次刷新后生效。
- 公共 KML 和个人 KML 使用相同的要素数据结构、渲染逻辑和坐标纠偏机制，避免重复实现。

## 前端改造要点

### KML 面板扩展

- `src/map/kml.js` 新增公共 KML 分区的 DOM 构建和事件处理。
- 公共 KML 渲染复用现有 `renderFeature`、`renderKmlLayers` 等函数，但需区分可编辑和只读模式。
- 新增 `loadSharedKmlList()` 和 `loadSharedKml(id)` 调用后端 API。
- 新增管理员身份检测逻辑。

### 后台新增 KML 管理页面

- `src/admin/pages/` 下新增 `kml.js` 面板模块。
- `src/admin/routes.js` 新增 KML 路由注册。
- `src/admin/state.js` 导航列表新增 KML 标签。
- `src/admin/api.js` 新增 KML 管理相关 API 调用方法。

### 后端改造

- `service/bin/admin/` 下新增 `sharedKml.js` 模块，管理公共 KML 的 CRUD 和文件上传解析。
- `service/bin/simpleApi.js` 注册新增的 KML 相关路由。
- 复用 AdminStore 进行数据持久化。
- KML XML 解析可提取 `src/map/kml-format.js` 中的 `parseKML` 为共享模块，或在服务端使用独立的 XML 解析实现。

## 验收标准

### 后台管理

- 后台 KML 管理页面能列出所有公共 KML，显示名称、状态、要素数量和时间信息。
- 后台可以导入 .kml 文件创建公共 KML。
- 后台可以新建空白公共 KML。
- 后台可以重命名公共 KML。
- 后台可以发布、禁用和删除公共 KML。
- 后台删除公共 KML 需要二次确认。
- 后台可以导出公共 KML 为标准 .kml 文件。
- 后台编辑内容按钮能跳转到前台地图并加载对应公共 KML。

### 管理员前台操作

- 管理员登录后，个人 KML 卡片显示"共享"按钮；未登录时不显示。
- 管理员点击共享按钮后，个人 KML 数据上传到服务端创建为公共 KML。
- 管理员从后台跳转到前台后，可以编辑公共 KML 要素（新增、删除、拖拽、编辑属性）。
- 管理员编辑完成后可以保存回服务端。
- 管理员未保存即关闭编辑模式时收到确认提示。

### 前台公共 KML

- 前台 KML 面板显示公共图层分区。
- 公共 KML 列表展示所有已发布的公共 KML 名称和要素数量。
- 用户可以启用公共 KML，启用后要素渲染到地图上。
- 用户可以禁用已启用的公共 KML，禁用后要素从地图移除。
- 用户的启用/禁用偏好在页面刷新后保持。
- 普通用户无法编辑公共 KML 内容（点位不可拖拽，无编辑/删除按钮）。
- 普通用户可以导出公共 KML。
- 公共 KML 列表可手动刷新。
- 后台禁用的公共 KML 在前台刷新后不再显示。
- 加载公共 KML 失败时显示错误提示，不影响其他功能。

### 工程标准

- 新增 API 位于 `/api/v1` 下，更新 `docs/api.md`。
- 前端业务代码不使用 `alert`、`confirm`、`prompt`。
- 不手动编辑 `service/app/` 构建产物。
- `npm run check`、`npm test`、`npm run build` 通过。
- 公共 KML 数据存储在 `.db/admin/shared-kml.json`，文档中说明存储位置。

## 分期建议

### 第一阶段：核心能力

- 后端公共 KML CRUD API。
- 后台 KML 管理页面（列表、新建、导入、重命名、发布、禁用、删除、导出）。
- 前台公共 KML 发现和加载。
- 前台管理员共享个人 KML。

### 第二阶段：编辑与发布

- 管理员前台编辑公共 KML 并保存回服务端。
- 从后台跳转到前台编辑公共 KML。
- 保存并发布工作流。
- 编辑未保存确认提示。

### 第三阶段：增强

- 公共 KML 排序和分组。
- 公共 KML 访问统计。
- 多管理员操作审计日志。
- 公共 KML 数据变更通知（用户侧提示有更新）。

## 与现有需求的关系

- 本需求是 [full-featured-map-application.md](full-featured-map-application.md) 中 F6（数据集与导入导出）和 F9（分享与协作）的子集落地。
- 数据模型与该文档中 A2（服务端数据模型）的 `MapDataset` 概念对齐，公共 KML 可视为 `visibility: 'public'` 的数据集。
- 后续如果实现完整的数据集体系，公共 KML 可以迁移为数据集的一种类型，API 路径可能变化，但核心数据结构兼容。
- 管理员身份识别复用 [admin-console-mvp.md](admin-console-mvp.md) 中的 Token 机制，不引入新的鉴权方案。
- 访问控制校验复用 [access-control-hardening.md](access-control-hardening.md) 中的 cookie 机制。
